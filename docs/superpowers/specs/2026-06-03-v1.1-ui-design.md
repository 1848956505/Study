# V1.1.0 项目整体 UI 设计

## 1. 目标

V1.1.0 的前端 UI 目标不是先把单个模块做满，而是先建立一套可长期扩展的工作台式界面框架，让后续知识库、试卷、AI、任务、复盘等模块都能在同一套壳层中自然生长。

本版 UI 的核心原则：

- 外层统一，内层分化
- 外层尽量薄，不抢占内容空间
- 模块页优先服务各自内容形态
- 知识库模块以内容阅读与写作为中心，尽量接近 Typora 的体验

## 2. 总体信息架构

界面分成两层：

### 2.1 全局工作台壳

全局壳层负责跨模块的高频能力，只承载这些内容：

- 模块切换
- 全局搜索
- 用户入口
- 设置入口
- 全局状态展示

全局壳层不负责承载模块的主要内容。

### 2.2 模块内工作区

各功能模块拥有自己的主工作区，并可根据场景调整布局：

- 知识库模块偏内容中心
- 试卷模块偏答题/审阅中心
- AI 模块偏对话与结果面板
- 任务模块偏列表与流转面板

## 3. 全局壳层布局

### 3.1 左侧导航

采用窄图标栏常驻模式：

- 默认只显示图标
- 鼠标悬停时显示功能名称
- 点击后切换模块

选择窄图标栏的原因是：

- 更省横向空间
- 不会压缩知识库正文
- 后续模块增多时仍然可控

### 3.2 顶部栏

顶部栏保持轻量，只放全局入口：

- 全局搜索
- 快捷创建/新增
- 设置
- 用户信息

顶部栏不做复杂信息堆叠，避免和模块主工具栏混在一起。

### 3.3 底部状态栏

底部状态栏用于显示系统状态：

- 保存状态
- 数据库/本地存储状态
- 同步状态
- AI 状态
- 快捷提示

它的定位是“感知层”，不是“操作层”。

不过底部状态栏也可以承载少量高频操作按钮，只保留那些不适合放在主工具栏、但又需要快速切换的能力，例如：

- AI 状态/AI 配置切换
- 当前工作模式切换
- 快速同步/刷新
- 本地存储状态相关快捷动作

原则是：

- 只放低频但高价值的操作
- 按钮数量保持克制
- 不要把底部栏变成第二个工具栏

## 4. 知识库模块 UI

知识库模块的默认形态是“内容中心”，整体风格更接近 Typora，而不是传统后台列表页。

### 4.1 默认布局

默认采用动态双栏/三栏结构，但主视觉重点始终放在正文区域：

- 左侧：目录、文件夹、资源树、笔记列表
- 中间：正文阅读与编辑主区
- 右侧：辅助信息区，如属性、关联、AI、引用、标签

原则上，正文区域永远优先获得最大空间。

### 4.2 动态展开

当用户进入特定操作场景时，辅助区域可以临时展开：

- 搜索时显示更强的筛选和结果信息
- 查看关联笔记时显示链接面板
- 使用 AI 辅助时展开右侧建议面板
- 进行批量管理时突出列表与操作区

这意味着知识库不是固定死板的三栏，而是一个可动态变形的工作区。

## 5. Markdown 编辑体验

### 5.1 默认模式

中间编辑区域默认直接显示格式化后的 Markdown 内容，不常驻源码编辑器。

目标体验接近 Typora：

- 用户看到的是排版后的内容
- 通过快捷键、按钮、浮动工具条完成格式操作
- 日常阅读和轻量编辑不需要频繁切换模式

### 5.2 源码编辑器的展开方式

保留一个“显示源码编辑器”的按钮。

点击后当前页面直接展开源码编辑器，并与预览区域并排显示：

- 左侧：Markdown 源码编辑
- 右侧：渲染预览

当前版本先实现这种并排模式，后续可扩展为整页源码模式，但不作为本期目标。

### 5.3 设计理由

这种方式兼顾两类用户：

- 习惯所见即所得的人，可以直接在预览中完成大部分操作
- 需要精修 Markdown 的人，可以随时打开源码面板

## 6. 视觉与交互原则

### 6.1 视觉基调

- 整体偏克制、清爽、专业
- 不使用过于重的后台感
- 主要空间留给内容本身
- 工具和状态信息尽量“退后一步”

### 6.2 交互原则

- 高频操作靠近内容区
- 低频设置收纳到全局壳层
- 默认状态尽量简洁
- 复杂操作按需展开，不长期占位

### 6.3 响应式原则

- 桌面端优先
- 窄图标栏应在中等宽度下保持可用
- 小屏幕下辅助栏和源码栏应优先折叠

## 7. 前端工程规范

为了让后续模块持续扩展而不失控，前端需要遵守统一的工程规范。

### 7.1 组件复用

- 组件优先复用，不重复造轮子
- 通用能力沉淀为基础组件或业务通用组件
- 业务页面只组合组件，不直接堆样式

### 7.2 样式管理

- 样式必须统一管理
- 尽量使用设计令牌、变量或统一主题层
- 颜色、间距、字号、圆角、阴影等不要散落硬编码
- 页面级样式只放少量局部例外

### 7.3 布局与交互一致性

- 相似模块使用相似布局规则
- 同类按钮、标签、状态提示保持一致样式
- 不同模块可以有自己的内容结构，但公共交互语言要统一

### 7.4 代码组织原则

- 业务逻辑、组件和样式尽量分层
- 可配置项优先配置化，不直接写死在页面里
- 通用状态和通用动作尽量抽到共享层

## 8. 设计边界

本版 UI 只解决“整体框架”和“知识库模块体验”的基础形态，不在本期强求：

- 完整的视觉品牌系统
- 全量动画效果
- 高级个性化主题
- 复杂的多窗口协作

先把壳层稳定下来，再让各模块逐步长出自己的体验。

## 9. 当前结论

本次讨论形成的明确结论是：

- 全局采用窄图标栏常驻
- 图标悬停显示名称
- 知识库模块默认偏 Typora 式内容中心
- 编辑器默认隐藏
- 点击按钮后源码编辑器与预览并排显示
- 未来可继续扩展为整页源码模式

这套方案适合当前“模块会越来越多”的产品方向，也能避免前端后续越做越重。
