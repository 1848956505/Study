# 学习加速器项目总控文档

## 0. 文档定位

本文档是“学习加速器”项目的长期总控文档，用于统一记录项目定位、核心目标、技术路线、模块规划、开发阶段、当前决策、暂不做范围以及后续沟通规则。

---

## 1. 项目名称

**学习加速器**（暂定）

可选副标题：

> 一个以知识库为核心，融合 AI 出题、智能批改、任务计划和阶段总结的个人学习辅助平台。

---

## 2. 项目定位

学习加速器是一个面向个人学习、项目成长与长期知识沉淀的智能辅助系统。

系统以 Markdown 知识库为核心，支持用户记录知识笔记、整理课程内容、沉淀学习资料，并通过 AI 能力实现知识总结、试卷生成、答案批改、错题分析、复习计划生成和阶段性周报总结。

项目不只是一个笔记软件，也不只是一个待办工具，而是一个围绕“知识输入—训练巩固—任务规划—阶段复盘”形成闭环的个人学习成长系统。

核心闭环：

```text
知识记录 → AI 理解 → 出题训练 → 批改反馈 → 计划安排 → 周报总结 → 再次沉淀
```

---

## 3. 核心目标

### 3.1 第一目标

构建一个可长期使用的个人学习平台，让用户能够把分散的笔记、课程、题目、计划和总结统一管理起来。

### 3.2 第二目标

通过 AI 将静态知识库转化为动态训练系统，使笔记不只是被收藏，而是能进一步生成题目、复习计划和学习反馈。

### 3.3 第三目标

建立一个可扩展的软件架构，后续可以逐步扩展为桌面端应用、移动端 App、B站课程总结、PDF 解析、网页收藏、知识图谱等能力。

---

## 4. 核心设计思想

### 4.1 知识库是大脑

知识库负责承载用户的 Markdown 笔记、课程总结、学习资料、项目记录和知识片段，是整个系统的基础数据来源。

### 4.2 试卷中心是训练场

试卷中心基于知识库内容生成题目，帮助用户通过做题检验掌握情况，并通过 AI 批改形成反馈。

### 4.3 修仙录是任务系统

修仙录本质上是待办、计划和成长记录模块，但采用游戏化包装，让学习任务更有仪式感和持续动力。

### 4.4 周报中心是成长记录

周报中心用于汇总用户一段时间内的学习内容、任务完成情况、试卷表现和项目进展，生成阶段性总结。

### 4.5 AI 是贯穿全局的辅助引擎

AI 不应只是一个孤立聊天框，而应嵌入知识库、试卷、批改、任务、周报等场景中，成为上下文感知的学习助手。


## 7. 推荐技术路线

### 7.1 总体路线

采用 Web 优先的全栈架构，后期再扩展到桌面端和移动端。

```text
第一阶段：Web 应用
第二阶段：PWA
第三阶段：Tauri / Electron 桌面端
第四阶段：Capacitor / Flutter 移动端
```

### 7.2 第一版推荐技术栈

| 层级 | 技术方案 | 说明 |
|---|---|---|
| 前端 | Next.js + React + TypeScript | Web 主应用 |
| UI | Tailwind CSS + shadcn/ui | 快速搭建现代化界面 |
| 后端 | NestJS + TypeScript | 业务 API 与模块化服务 |
| ORM | Prisma | 数据库模型与查询 |
| 数据库 | PostgreSQL | 主业务数据存储 |
| 向量检索 | pgvector | 知识库 RAG 与相似度检索 |
| 缓存/队列 | Redis + BullMQ | AI 长任务、异步任务 |
| 文件存储 | MinIO / S3 | 图片、附件、导出文件 |
| AI 服务 | 独立 AI Service | 统一封装模型调用 |
| 桌面端 | Tauri / Electron | 后期封装 Web 应用 |
| 移动端 | PWA / Capacitor / Flutter | 后期移动端扩展 |

### 7.3 技术原则

1. Web 优先，不一开始做多端。
2. 模块化单体优先，不一开始做微服务。
3. API 优先，保证后续多端共用。
4. AI 服务统一封装，不在业务代码中直接散落模型调用。
5. 数据模型先稳定，再开发上层功能。
6. 长任务走队列，避免前端长时间阻塞。
7. 第一版先跑通闭环，再逐步扩展。
8. 功能以模块化方式呈现，高内聚低耦合。
9. 前端遵循构建设计系统，即遵循设计令牌、原子设计、组件库 + 样式库的工程化分离

## 10. AI 能力设计原则

### 10.1 AI 能力统一入口

所有 AI 调用统一经过 AI Service，不允许业务模块直接调用具体模型供应商。

推荐结构：

```text
业务模块
    ↓
AI Service
    ↓
Model Provider Adapter
    ↓
OpenAI / DeepSeek / Qwen / Claude / 本地模型
```

### 10.2 AI 输出尽量结构化

对于试卷生成、批改结果、错题分析等功能，AI 应尽量输出 JSON，后端进行结构校验后再入库。

### 10.3 RAG 优先基于知识库

AI 出题、问答和总结应优先基于用户知识库内容，不应无依据扩写。

### 10.4 重要 AI 任务应保存记录

需要保存：

- 输入内容
- 检索上下文
- 使用模型
- 提示词版本
- 输出结果
- 执行状态
- 错误信息

这样后续才能追踪、复盘和优化。
