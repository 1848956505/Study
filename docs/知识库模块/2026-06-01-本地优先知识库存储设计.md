# 学习加速器 V1.0 本地优先知识库存储设计

## 1. 文档目标

本文档定义 `学习加速器` V1.0 在 `本地优先` 场景下的知识库存储方案。

这里的“本地优先”指：

- 用户的核心知识数据默认保存在用户自己的设备或本机运行的服务中
- 系统在 V1.0 不依赖外部云数据库才能工作
- 后续如需多设备同步或联网访问，可以在不改动核心领域模型的前提下迁移到云端部署

---

## 2. 设计结论

V1.0 推荐采用以下本地优先存储方案：

- `PostgreSQL` 作为本地核心数据库
- `pgvector` 作为本地语义检索向量扩展
- `Redis` 作为本地缓存与任务队列支撑
- `本地文件系统` 或 `MinIO` 作为附件与导出文件存储

这意味着：

- 笔记正文、标签、目录、切片、向量、任务记录都保存在本机
- 云端数据库不是 V1.0 的前置依赖
- 系统优先保证“单机可运行、单机可恢复、单机可演进”

---

## 3. 为什么选择本地 PostgreSQL

### 3.1 不建议只用 Markdown 文件保存

如果只把知识库存成 Markdown 文件，短期看简单，但很快会遇到问题：

- 文件夹、标签、多条件搜索不容易稳定管理
- 文档切片与向量索引难以维护
- AI 任务记录无法统一追踪
- 删除恢复、索引重建、后续扩展都容易混乱

因此，Markdown 文件可以作为导出格式或导入来源，但不建议作为系统唯一数据源。

### 3.2 不建议一开始依赖云数据库

如果 V1.0 一开始就强依赖云数据库，会增加：

- 部署复杂度
- 维护成本
- 网络依赖
- 用户数据敏感性顾虑

对个人学习产品来说，先把数据留在本机，更符合早期使用习惯。

### 3.3 PostgreSQL 适合作为本地优先底座

原因：

- 支持结构化关系模型
- 适合复杂查询与后续扩展
- 可直接搭配 `pgvector`
- 适合保存 AI 任务记录和搜索相关数据
- 后期迁移到云端 PostgreSQL 成本低

---

## 4. 本地优先存储分层

### 4.1 核心业务数据

保存位置：`本地 PostgreSQL`

包含：

- 用户
- 用户设置
- 知识空间
- 文件夹
- 笔记
- 标签
- 笔记标签关系
- 笔记引用关系
- 文档切片
- AI 任务记录
- 索引任务记录

### 4.2 语义检索数据

保存位置：`本地 PostgreSQL + pgvector`

包含：

- 切片向量
- 向量模型信息
- 向量版本信息
- 向量状态

### 4.3 缓存和任务状态

保存位置：`本地 Redis`

包含：

- 异步任务队列
- 短期缓存
- 索引构建任务状态
- AI 总结任务状态

说明：

- Redis 不是长期真源
- Redis 数据可以丢失后重建

### 4.4 附件和导出物

保存位置：`本地文件系统`，后续可切换到 `MinIO`

包含：

- 图片附件
- 导出文件
- 周报 PDF
- 其他静态资源

V1.0 建议优先用本地目录，例如：

```text
storage/
├── uploads/
├── exports/
└── temp/
```

## 5. 本地优先下的数据组织策略

### 5.1 原始数据与派生数据分离

必须区分：

- 原始数据：用户编辑的 Markdown 内容
- 派生数据：纯文本、摘要、切片、向量、搜索索引

推荐规则：

- `Note.raw_markdown` 是唯一可信原文
- `Note.plain_text` 是派生结果
- `DocumentChunk` 是派生结果
- `Embedding` 是派生结果

这意味着即使索引损坏，也可以从原始笔记重建。

### 5.2 单机单用户优先，但保留多用户结构

虽然本地优先常常意味着个人使用，但模型设计仍建议保留：

- `User`
- `KnowledgeSpace`
- 数据按用户隔离

这样后续迁移到云端时不需要重构领域模型。

### 5.3 路径与数据库解耦

即使附件放在本地文件夹中，也不建议只靠目录结构组织全部信息。

正确做法是：

- 文件实际保存在本地目录
- 文件元数据保存在数据库中
- 数据库记录文件逻辑归属

这样更利于迁移、重命名和恢复。

---

## 6. 推荐部署形态

### 6.1 形态 A：开发阶段本地服务

适用场景：

- 项目开发
- 单机调试
- 验证知识库能力

组成：

- Web 前端本地运行
- API 服务本地运行
- PostgreSQL 本地运行
- Redis 本地运行
- 附件写入本地目录

这是 V1.0 最推荐的起步方式。

### 6.2 形态 B：个人正式使用的本机部署

适用场景：

- 用户个人长期使用
- 不依赖外网
- 不要求跨设备同步

组成：

- 本机 API 服务
- 本机数据库
- 本机存储目录
- 可选桌面封装壳层

### 6.3 形态 C：未来云端迁移

适用场景：

- 多设备访问
- 远程备份
- 多端同步

迁移方式：

- 保持应用层接口不变
- 将本地 PostgreSQL 替换为云端 PostgreSQL
- 将本地文件存储替换为对象存储
- Redis 保持同类替换

本地优先方案的价值就在于：迁移的是部署位置，不是业务模型。

---

## 7. 数据备份与恢复策略

本地优先必须把备份考虑进去，否则设备损坏会直接丢数据。

### 7.1 必须备份的数据

- PostgreSQL 数据库
- 本地附件目录
- 导出文件目录

### 7.2 可重建的数据

以下数据理论上可以从原始笔记重建：

- `plain_text`
- `DocumentChunk`
- `Embedding`
- 搜索索引

因此备份优先级排序建议是：

1. 原始数据库
2. 本地附件
3. 派生索引

### 7.3 V1.0 备份建议

V1.0 可以先做到：

- 提供数据库导出说明
- 提供 `storage/` 目录备份说明
- 预留后续“一键备份”功能

---

## 8. 本地优先下的知识库生命周期

### 8.1 新建笔记

1. 用户提交 Markdown 内容
2. API 写入本地 PostgreSQL
3. 生成 `plain_text`
4. 投递本地切片任务
5. 投递本地向量任务

### 8.2 编辑笔记

1. 更新 `raw_markdown`
2. 更新 `plain_text`
3. 标记旧切片过期
4. 重新生成切片
5. 重新生成向量

### 8.3 删除笔记

1. 软删除 `Note`
2. 清理标签关系
3. 清理或标记切片失效
4. 清理搜索数据

### 8.4 索引异常恢复

如果出现：

- 切片失败
- 向量丢失
- 搜索不一致

系统应支持：

- 基于 `Note.raw_markdown` 重新构建派生数据

---

## 9. 本地优先的风险与应对

### 9.1 风险：用户不会维护数据库

应对：

- 用 Docker 或脚本降低启动难度
- 提供明确的开发与运行文档
- 后续可考虑桌面安装器或一体化启动器

### 9.2 风险：设备损坏导致数据丢失

应对：

- 提供备份建议
- 后续增加导出与自动备份能力

### 9.3 风险：本地资源占用增加

应对：

- 限制向量重建频率
- 异步处理长任务
- 清理临时文件

### 9.4 风险：后续迁移复杂

应对：

- 坚持数据库优先，而不是文件优先
- 让存储位置可配置
- 不把本地路径写死在业务层

---

## 10. 设计建议总结

V1.0 的本地优先知识库存储，推荐采用如下组合：

- `PostgreSQL`：核心知识数据真源
- `pgvector`：语义检索向量
- `Redis`：缓存与队列
- `storage/` 本地目录：附件与导出文件

最终原则是：

- 核心知识数据存在数据库
- 派生索引可以重建
- 文件存储独立于数据库但受数据库管理
- 本地可运行，后续可迁移

这套方案既适合 V1.0 的个人学习产品起步，也能为未来的云端版本保留足够平滑的演进路径。
