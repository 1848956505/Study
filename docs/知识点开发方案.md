# 资料库知识点功能设计文档

## 0. 文档定位

本文档用于描述「学习加速器」中资料库（笔记功能）内的知识点功能设计。

本文档只讨论资料库中当前笔记页面内的知识点创建、展示、编辑、筛选、定位与管理逻辑；不展开讨论独立「知识库中心」、试卷中心、AI 出题页面、知识图谱页面等后续模块的具体页面设计。

知识点功能的核心定位是：

> **正文原文证据 + 注释 + 标签体系 + 可定位锚点 + 后续 AI 出题 / 知识图谱 / GraphRAG 的最小知识单元。**

其中「注释」统一指用户对该知识点的个人理解、感悟、解释、评价或补充说明。文档后文统一使用「注释」这一名称，不再混用「个人理解」「解释」「评论」等字段名。

该功能不是普通摘抄，也不是单纯标签，而是把笔记正文中的关键内容转化为可管理、可复习、可检索、可出题、可建图的结构化学习对象。

---

## 一、功能边界与核心原则

### 1. 功能边界

本功能属于资料库的笔记编辑 / 阅读场景，主要服务于当前笔记页面。

当前阶段重点包括：

1. 在笔记正文中选中文本创建知识点。
2. 将选中文本加入已有知识点。
3. 在右侧「知识点」面板中管理当前笔记相关知识点。
4. 支持知识点标题、注释、标签、原文片段的增删改查。
5. 支持正文原文片段与右侧知识点卡片之间的双向定位。
6. 为后续 AI 出题、复习统计、错题关联、知识图谱和 GraphRAG 建立稳定的数据基础。

当前阶段不展开实现：

1. 独立知识库中心页面。
2. 试卷中心页面。
3. AI 自动出题完整流程。
4. 知识图谱可视化页面。
5. GraphRAG 检索流程。

但是，当前知识点的数据结构与交互设计必须为这些后续功能预留扩展空间。

### 2. 核心原则

知识点功能应遵循以下原则：

1. **原文可追溯**：每个知识点至少绑定一条原文片段，原文片段必须能够回到笔记中的来源位置。
2. **用户可解释**：知识点不仅保存原文，还应允许用户写下自己的注释。
3. **标签可组织**：知识点通过普通标签和系统预定义标签组进行筛选、复习和后续出题。
4. **结构可扩展**：知识点不是一次性 UI 数据，而是未来 AI 出题、错题分析、掌握情况统计、知识图谱构建的基础实体。
5. **前端轻操作**：用户在读笔记时应能顺手创建知识点，不应被复杂表单打断。
6. **后端强约束**：前端可以灵活展示，但关键规则必须由后端兜底，例如必填字段、标签互斥规则、用户数据隔离、原文片段归属关系等。

## 二、知识点的产品数据结构

从产品视角看，一个知识点至少包含以下内容：

```text
知识点
├── 标题：必选
├── 注释：可选
├── 原文片段：必选，至少一条
├── 普通标签：可选，可多选
├── 系统预定义标签组：可选或有默认值
│   ├── 掌握程度：单选
│   ├── 重要程度：单选
│   └── 用途：多选
├── 来源笔记：自动关联
├── 正文定位锚点：自动生成
└── 创建时间 / 更新时间：自动记录
```

### 1. 标题

标题是知识点的最短识别名称，也是未来复习、出题、知识图谱节点展示时最重要的入口。

标题为必填字段。

创建知识点时，标题可以默认显示为：

```text
未命名知识点
```

保存时建议要求用户填写有效标题。也可以在 MVP 阶段允许保存「未命名知识点」，但前端应给出明显提示，后续在知识点列表中提醒补全。

### 2. 注释

注释用于记录用户对知识点的个人理解、感悟、解释、评价或补充说明。

注释为可选字段。

注释不是 AI 总结字段，也不是原文复刻字段。它更接近用户自己的「消化结果」。后续 AI 出题或复习时，可以同时参考原文片段与用户注释。

### 3. 原文片段

原文片段是知识点的证据来源。

每个知识点至少需要一条原文片段。没有原文片段的知识点可以作为草稿存在，但不能作为正式知识点保存。

一个知识点可以包含多个原文片段，用于处理以下情况：

1. 定义在笔记前部，例子在笔记中部。
2. 一个知识点由多段分散文本共同构成。
3. 用户后续阅读时发现新的补充内容，需要加入已有知识点。

### 4. 标签

标签分为两类：

```text
普通标签
系统预定义标签组
```

普通标签主要用于自由分类；系统预定义标签组用于承载具有固定语义和固定规则的学习状态信息。

## 三、右侧知识点面板整体结构

右侧「知识点」Tab 是当前笔记内知识点管理的主要入口。

建议整体分为 4 层：

```text
顶部操作区
筛选搜索区
知识点卡片列表区
底部批量操作区
```

### 1. 顶部操作区

顶部操作区用于展示当前笔记知识点数量和常用操作。

推荐结构：

```text
知识点  6
[+] [筛选] [更多]
```

含义：

- `知识点 6`：当前笔记关联的知识点数量。
- `+`：手动创建知识点草稿。
- `筛选`：展开或收起筛选搜索区。
- `更多`：显示批量管理、显示设置、导出等扩展操作。

如果当前正文没有选中文本，点击 `+` 可以打开空白草稿卡片，但保存时必须要求至少添加一条原文片段。

提示文案建议为：

```text
请先从正文中选中文本，或在稍后为该知识点添加原文片段。
```

保存校验文案建议为：

```text
知识点至少需要绑定一条原文片段后才能保存。
```

### 2. 筛选搜索区

知识点数量变多后，筛选搜索区必须能够帮助用户快速定位知识点。

推荐包含：

```text
搜索框：搜索标题、注释、原文片段
普通标签筛选：多模态 / 英语 / 自动机 / 计网 / CLIP
掌握程度：全部 / 未掌握 / 模糊 / 已掌握
重要程度：全部 / 重点 / 一般 / 可跳过
用途：全部 / 可出题 / 需背诵 / 需理解 / 易错
排序方式：按正文位置 / 按更新时间 / 按重要程度
```

这里需要注意：虽然底层都可以通过 tag 体系实现，但前端展示时不应该把所有标签混在一个列表中，而应该按照语义分组展示。

推荐视觉分组：

```text
普通标签：多模态、英语、计网、自动机、CLIP
掌握程度：未掌握、模糊、已掌握
重要程度：重点、一般、可跳过
用途：可出题、需背诵、需理解、易错
```

后续文档统一使用「普通标签」这一名称。

## 四、知识点卡片设计

知识点以卡片形式展示在右侧「知识点」面板中。

卡片建议采用「默认折叠，点击展开」的形式，避免右侧面板过于拥挤。

### 1. 默认卡片形态

```text
┌─────────────────────────────┐
│ CLIP 对比学习损失             │
│ #多模态  #CLIP               │
│ 掌握：未掌握  重要：重点       │
│ 用途：可出题 / 需理解          │
│                              │
│ 注释：图文对比学习的核心是...   │
│                              │
│ 原文 2 段              [展开] │
└─────────────────────────────┘
```

默认卡片只显示：

1. 知识点标题。
2. 普通标签。
3. 系统预定义标签组中的关键状态。
4. 注释摘要。
5. 原文片段数量。
6. 展开、定位、编辑、更多操作入口。

### 2. 展开卡片形态

```text
┌─────────────────────────────┐
│ CLIP 对比学习损失             │
│ #多模态  #CLIP               │
│ 掌握：未掌握  重要：重点       │
│ 用途：可出题 / 需理解          │
│                              │
│ 注释：                        │
│ 图文对比学习的核心是让匹配的图文对... │
│                              │
│ 原文片段                      │
│ 1. “CLIP uses a contrastive...”│
│    [定位] [删除片段]           │
│                              │
│ 2. “The loss is symmetric...” │
│    [定位] [删除片段]           │
│                              │
│ [添加原文片段] [编辑知识点]     │
└─────────────────────────────┘
```

展开后重点展示三件事：

1. 用户如何理解这个知识点。
2. 这个知识点来自哪些原文片段。
3. 用户可以如何继续管理这个知识点。

## 五、从正文创建知识点的交互流程

### 流程 1：选中文本后创建新知识点

用户在 Markdown 正文中选中文本后，鼠标右击出现快捷功能菜单。

快捷功能菜单中包含「知识点」子菜单：

```text
知识点
├── 创建知识点
└── 加入已有知识点
```

点击「创建知识点」后：

1. 右侧面板自动切换到「知识点」Tab。
2. 新建一个知识点草稿卡片。
3. 用户刚刚选中的文本自动成为第一条原文片段。

草稿卡片结构：

```text
标题：_________
普通标签：
[+ 添加普通标签]

掌握程度：未掌握 / 模糊 / 已掌握
重要程度：重点 / 一般 / 可跳过
用途：可出题 / 需背诵 / 需理解 / 易错

注释：_________

原文片段：
> 用户刚刚选中的文本

[保存] [取消]
```

标题默认建议使用：

```text
未命名知识点
```

也可以根据当前 Markdown 小标题或选中文本前 10～20 个字生成临时标题，但应允许用户修改。

### 流程 2：多段文本创建同一个知识点

多段文本应通过「采集模式」实现。

用户开启采集模式后，可以在正文中多次选择文本，并将这些文本连续加入同一个知识点草稿。

交互示例：

```text
开启采集模式
↓
选择第一段文本 → 加入当前知识点草稿
↓
选择第二段文本 → 加入当前知识点草稿
↓
选择第三段文本 → 加入当前知识点草稿
↓
保存知识点
```

右侧草稿卡片显示：

```text
原文片段 3 段
1. ...
2. ...
3. ...
```

该模式适合处理定义、例子、注意事项分散在不同位置的情况。

### 流程 3：把新文本加入已有知识点

用户选中文本后点击：

```text
加入已有知识点
```

弹出小型搜索框：

```text
搜索知识点标题 / 普通标签 / 系统标签 / 原文片段
```

搜索结果优先展示：

```text
当前笔记中的知识点
```

再展示：

```text
当前知识空间中的其他知识点
```

这里仍然属于资料库笔记页面内的交互入口。只是由于知识点本身未来会作为全局对象存在，所以前端可以预留跨笔记加入已有知识点的能力。MVP 阶段可以先只支持当前笔记内知识点，后续再扩展到当前知识空间或全局知识点。

选择某个知识点后，弹出确认：

```text
已选择文本将作为新的原文片段加入：
「CLIP 对比学习损失」

[确认加入] [取消]
```

确认后：

1. 新原文片段加入该知识点。
2. 右侧对应卡片自动展开。
3. 新增原文片段短暂高亮。
4. 正文中该片段显示知识点标记。

## 六、正文区域的知识点标记方式

右侧卡片不能孤立存在，必须与正文形成双向联动。

### 1. 正文轻量标记

被加入知识点的原文，在正文中显示轻量标记。

推荐以轻微下划线为主，必要时配合浅色背景或左侧小标记。

```text
默认：轻微下划线
可选：浅色背景 / 左侧小标记
```

不建议使用过重颜色，否则会破坏 Markdown 阅读和编辑体验。

提供开关：

```text
显示知识点标记：开 / 关
```

默认开启。用户专注写作时可以关闭。

### 2. 鼠标悬停提示

鼠标移动到被标记的文本上时，出现小浮层：

```text
知识点：CLIP 对比学习损失
普通标签：#多模态 #CLIP
掌握：未掌握
重要：重点

[打开卡片] [编辑]
```

该浮层用于让用户快速知道：这段正文已经被收进哪个知识点。

### 3. 点击正文标记定位卡片

用户点击正文中被标记的文本：

```text
正文标记 → 右侧自动切换到知识点 Tab → 对应卡片滚动到可见区域 → 卡片高亮 1 秒
```

### 4. 点击卡片定位原文

用户点击知识点卡片中的「定位」按钮：

```text
点击定位 → 编辑器滚动到原文位置 → 原文片段闪烁高亮
```

如果一个知识点有多个原文片段，则每个原文片段都应有独立的定位按钮。

## 七、知识点编辑交互

知识点卡片支持三种状态：

```text
浏览态
快速编辑态
完整编辑态
```

### 1. 浏览态

默认只展示知识点内容，不进入表单编辑。

### 2. 快速编辑态

点击局部内容即可快速编辑：

```text
标题点击 → 变成输入框
普通标签点击 → 打开普通标签选择器
掌握程度点击 → 打开单选状态菜单
重要程度点击 → 打开单选状态菜单
用途点击 → 打开多选菜单
注释点击 → 变成小文本框
```

快速编辑适合轻量修改，不应打断阅读。

### 3. 完整编辑态

点击「编辑知识点」后，卡片进入完整表单状态，或打开右侧面板内的编辑区域。

推荐表单结构：

```text
标题
普通标签
掌握程度
重要程度
用途
注释
原文片段列表
```

不建议使用居中弹窗，因为居中弹窗会遮挡正文，不利于对照原文修改。

## 八、标签系统与系统预定义标签组

标签系统是知识点后续用于筛选、复习、出题和构建知识图谱的关键基础。

本功能中，标签统一分为两类：

```text
普通标签
系统预定义标签组
```

### 1. 普通标签

普通标签用于自由分类。

示例：

```text
多模态、英语、自动机、计网、CLIP、Transformer、操作系统、数据结构
```

普通标签特点：

1. 用户可以自由创建。
2. 一个知识点可以绑定多个普通标签。
3. 普通标签之间一般不互斥。
4. 普通标签主要服务于分类、搜索、筛选、出题范围选择和知识图谱聚类。

### 2. 系统预定义标签组

系统预定义标签组是系统预先定义好的特殊标签集合。

它们底层仍然可以作为标签存储，但前端和后端都必须知道它们具有固定语义和固定规则。

推荐系统预定义标签组如下：

```text
掌握程度：未掌握 / 模糊 / 已掌握
重要程度：重点 / 一般 / 可跳过
用途：可出题 / 需背诵 / 需理解 / 易错
```

其中：

1. 掌握程度：单选，一个知识点只能有一个掌握状态。
2. 重要程度：单选，一个知识点只能有一个重要程度。
3. 用途：多选，一个知识点可以同时是「可出题」「需背诵」「易错」。

### 3. 为什么要区分普通标签和系统预定义标签组

虽然底层可以统一使用 tag 体系，但它们在业务语义上并不相同。

普通标签是自由分类，例如：

```text
#多模态 #CLIP #对比学习
```

这些标签可以同时存在。

但掌握程度不能同时出现：

```text
#未掌握 #已掌握
```

这种组合在业务上自相矛盾。因此，掌握程度必须作为单选系统预定义标签组处理。

重要程度也同理，一个知识点不应同时是「重点」和「可跳过」。

### 4. 标签选择器设计

标签选择器建议采用分组展示：

```text
添加标签

普通标签
[多模态] [英语] [自动机] [计网] [CLIP]
[+ 新建普通标签]

掌握程度
[未掌握] [模糊] [已掌握]

重要程度
[重点] [一般] [可跳过]

用途
[可出题] [需背诵] [需理解] [易错]
```

视觉上可以区分：

1. 普通标签：常规标签样式。
2. 掌握程度：状态胶囊。
3. 重要程度：强调程度样式。
4. 用途：功能标签样式。

但不建议只靠颜色传递语义，应该配合文字。

## 九、删除与移除要区分

知识点未来会作为全局对象存在，因此在当前笔记页面中，必须区分「从当前笔记移除」和「删除知识点」。

```text
从当前笔记移除
删除知识点
```

### 1. 从当前笔记移除

含义：

1. 取消该知识点与当前笔记的关联。
2. 移除该知识点在当前笔记中的原文片段标记。
3. 不删除知识点本体。
4. 不影响该知识点与其他笔记的关联。

适合场景：用户认为当前笔记不应再关联该知识点，但知识点本身仍有价值。

### 2. 删除知识点

含义：

1. 删除知识点本体。
2. 删除其注释、标签关系和原文片段关系。
3. 该知识点不再参与复习、出题、知识图谱和 GraphRAG。

前端确认文案建议：

```text
删除后，该知识点将不再出现在复习、出题和知识图谱中。该操作会影响所有关联笔记，确认删除吗？
```

如果系统支持回收站或软删除，则可以显示：

```text
删除后可在回收站中恢复。
```

## 十、右侧面板的状态设计

### 1. 空状态

当前笔记没有知识点时：

```text
还没有知识点

选中正文中的关键内容，即可创建知识点。
知识点可用于后续 AI 出题、复习和知识图谱构建。

[创建知识点]
```

### 2. 有选中文本但未创建

当用户选中了正文但还没有操作时，右侧可以轻提示：

```text
已选中一段文本，可创建知识点。

[创建知识点] [加入已有]
```

### 3. 无搜索结果

```text
没有找到匹配的知识点

可以换个关键词，或清除筛选条件。

[清除筛选]
```

### 4. 原文位置失效

如果笔记正文被修改，原来的定位锚点找不到了，卡片中显示：

```text
原文位置可能已变化
仍保留原文快照，但无法准确定位到正文。

[尝试重新定位] [手动选择新位置]
```

该状态非常重要。因为知识点必须保留原文快照，不能因为正文修改就丢失知识点证据。

## 十一、推荐的前端组件拆分

推荐组件拆分如下：

```text
KnowledgePointPanel
```

3. `KnowledgePointFilterBar`：搜索、普通标签筛选、系统预定义标签组筛选。
4. `KnowledgePointCardList`：知识点卡片列表。

其中最关键的是：

```text
SelectionContextMenu
EditorKnowledgeHighlightLayer
```

前者决定创建知识点是否顺手；后者决定正文与知识点是否能形成一体化体验。

## 十二、推荐的后端设计

本节是推荐设计，不是强制实现方案。由于具体项目代码结构、数据库模型、ORM 写法、模块边界需要结合现有代码判断，Codex 在实现时应根据项目实际情况进行调整。

但是无论具体代码如何实现，都应满足以下后端设计原则：

1. 可扩展性优先。
2. 知识点应作为长期核心实体设计，而不是只服务当前右侧面板的临时数据。
3. 标签体系必须支持普通标签与系统预定义标签组。
4. 原文片段必须保存快照与定位信息。
5. 后端必须保证用户 / 知识空间数据隔离。
6. 所有关键操作应便于后续接入 AI 出题、复习统计、错题分析、知识图谱和 GraphRAG。

### 1. 推荐模块边界

推荐将知识点能力放在知识库相关模块中，例如：

```text
Knowledge Module
├── Note Service
├── Folder Service
├── Tag Service
├── KnowledgePoint Service
├── SourceSnippet Service
└── KnowledgePointQuery Service
```

如果项目当前已经有知识库模块，则优先在现有模块内扩展，不建议为了知识点功能单独拆成过重的新服务。

如果后续知识点功能变复杂，再考虑在模块内部进一步拆分 service，而不是一开始就做微服务。

### 2. 推荐核心实体

推荐从以下实体方向设计。具体表名、字段名可由 Codex 根据项目规范调整。

#### 2.1 KnowledgePoint

表示知识点本体。

推荐字段方向：

```text
id
userId
spaceId
title
comment
status
createdAt
updatedAt
deletedAt
```

说明：

1. `title`：知识点标题，必填。
2. `comment`：注释，可选。
3. `status`：可选，用于草稿、正常、归档等状态。
4. `deletedAt`：推荐支持软删除，便于恢复和防止误删。
5. `userId`、`spaceId`：用于数据隔离和知识空间筛选。

#### 2.2 KnowledgePointSource

表示知识点绑定的原文片段。

推荐字段方向：

```text
id
knowledgePointId
noteId
spaceId
sourceText
sourceTextHash
anchor
startOffset
endOffset
blockId
noteVersion
contextBefore
contextAfter
sortOrder
isAnchorValid
createdAt
updatedAt
```

说明：

1. `sourceText`：保存用户选中的原文快照。
2. `sourceTextHash`：用于后续检测文本是否变化。
3. `anchor`：保存编辑器可识别的定位锚点信息，可以是 JSON。
4. `startOffset` / `endOffset`：保存偏移量，适合纯文本定位。
5. `blockId`：如果编辑器支持块级 ID，可用于更稳定定位。
6. `noteVersion`：记录创建片段时的笔记版本，便于判断定位是否失效。
7. `contextBefore` / `contextAfter`：保存前后少量上下文，便于锚点失效后重新定位。
8. `isAnchorValid`：标记当前锚点是否仍然有效。

原文片段设计不要只保存起止位置。因为笔记会频繁修改，单纯位置很容易失效。必须保存原文快照和必要上下文，后续才能做重新定位。

#### 2.3 Tag

表示标签本体。

推荐字段方向：

```text
id
userId
spaceId
groupId
name
code
color
isSystem
isArchived
createdAt
updatedAt
```

说明：

1. 普通标签可以由用户创建。
2. 系统预定义标签也可以复用 Tag 表，但应通过 `isSystem`、`groupId`、`code` 等字段区分。
3. 不建议只靠标签中文名判断业务逻辑，应该使用稳定的 `code`。

#### 2.4 TagGroup

表示标签组，用于支持普通标签和系统预定义标签组。

推荐字段方向：

```text
id
spaceId
name
code
type
selectionMode
isSystem
sortOrder
createdAt
updatedAt
```

推荐内置标签组：

```text
ordinary：普通标签，multiple
mastery：掌握程度，single
importance：重要程度，single
purpose：用途，multiple
```

说明：

1. `selectionMode` 用于约束该组是单选还是多选。
2. 掌握程度、重要程度应设置为 `single`。
3. 用途和普通标签应设置为 `multiple`。
4. 后端应根据 `selectionMode` 校验同一知识点在同一标签组下的绑定数量。

#### 2.5 KnowledgePointTag

表示知识点与标签之间的关系。

推荐字段方向：

```text
knowledgePointId
tagId
createdAt
```

说明：

1. 用于支持一个知识点绑定多个标签。
2. 后端保存时应校验系统预定义标签组的互斥规则。
3. 查询时应支持按普通标签、掌握程度、重要程度、用途组合筛选。

#### 2.6 NoteKnowledgePoint

表示笔记与知识点之间的关联。

推荐字段方向：

```text
noteId
knowledgePointId
spaceId
createdAt
updatedAt
```

说明：

1. 当前笔记右侧面板查询的是该笔记关联的知识点。
2. 一个知识点可以关联多个笔记。
3. 一个笔记可以关联多个知识点。
4. 从当前笔记移除知识点时，应优先删除或禁用这层关联，而不是直接删除 KnowledgePoint。

如果 KnowledgePointSource 已经足够表达 noteId 与 knowledgePointId 的关联，Codex 可以根据现有代码判断是否仍需要单独的 NoteKnowledgePoint 表。推荐保留独立关联表，因为它更利于后续全局知识点管理、批量关联和统计。

### 3. 推荐接口能力

以下为推荐接口能力，不要求固定路径名称。

#### 3.1 当前笔记知识点查询

用途：右侧知识点面板初始化。

能力：

```text
根据 noteId 查询当前笔记关联的知识点
支持搜索标题、注释、原文片段
支持普通标签筛选
支持掌握程度筛选
支持重要程度筛选
支持用途筛选
支持排序：正文位置 / 更新时间 / 重要程度
```

#### 3.2 创建知识点

用途：选中文本后创建知识点。

能力：

```text
创建 KnowledgePoint
创建至少一条 KnowledgePointSource
绑定普通标签和系统预定义标签
建立 NoteKnowledgePoint 关联
```

后端校验：

```text
标题不能为空
正式保存时原文片段不能为空
原文片段必须属于当前用户可访问的 noteId
系统预定义标签组必须满足 single / multiple 规则
```

#### 3.3 更新知识点

用途：编辑标题、注释、标签状态。

能力：

```text
更新标题
更新注释
更新普通标签
更新掌握程度
更新重要程度
更新用途
```

后端应支持局部更新，避免前端每次都提交完整对象。

#### 3.4 添加原文片段

用途：将新文本加入已有知识点。

能力：

```text
向已有 KnowledgePoint 添加新的 KnowledgePointSource
必要时建立当前 noteId 与 KnowledgePoint 的关联
返回更新后的知识点卡片数据
```

#### 3.5 删除原文片段

用途：在展开卡片中删除某一条原文片段。

注意：如果删除后该知识点没有任何原文片段，后端应阻止删除，或将知识点转为草稿状态。

推荐提示：

```text
知识点至少需要保留一条原文片段。
```

#### 3.6 从当前笔记移除知识点

用途：取消当前笔记与知识点的关系。

能力：

```text
删除或禁用 NoteKnowledgePoint 关联
移除当前 noteId 下相关 KnowledgePointSource 标记
不删除 KnowledgePoint 本体
```

如果该知识点只关联当前笔记，后端可以选择保留为未关联知识点，或提示用户是否彻底删除。MVP 阶段建议简单处理：只移除当前笔记关联，不自动删除知识点本体。

#### 3.7 删除知识点

用途：彻底删除知识点。

推荐使用软删除：

```text
设置 KnowledgePoint.deletedAt
同时隐藏相关 source、tag relation、note relation
```

不建议直接硬删除，除非用户明确执行永久删除。

#### 3.8 标签与标签组查询

用途：前端初始化标签选择器和筛选器。

能力：

```text
查询普通标签
查询系统预定义标签组
查询每个标签组下的标签
返回 selectionMode、isSystem、sortOrder 等信息
```

前端不应该硬编码所有系统标签。前端可以有默认展示逻辑，但标签组配置应以后端返回为准。

### 4. 后端扩展性规范

后端实现时尤其要注意以下规范。

#### 4.1 不要把知识点做成笔记的附属 JSON 字段

不推荐：

```text
Note.knowledgePoints = JSON
```

原因：

1. 后续无法方便按标签生成试卷。
2. 无法跨笔记复用知识点。
3. 无法做知识图谱节点。
4. 无法进行稳定统计和筛选。
5. 不利于 GraphRAG 和向量检索。

推荐：

```text
KnowledgePoint 作为独立实体
SourceSnippet / Tag / NoteRelation 作为关联实体
```

#### 4.2 不要把掌握程度、重要程度写死成普通字符串

不推荐：

```text
knowledgePoint.mastery = "未掌握"
knowledgePoint.importance = "重点"
```

推荐：

```text
通过 TagGroup + Tag + KnowledgePointTag 表达
```

原因：

1. 可以统一筛选逻辑。
2. 可以扩展新的系统预定义标签组。
3. 可以支持用户自定义展示名称。
4. 可以支持多语言和排序。
5. 可以在后端统一校验单选 / 多选规则。

如果 Codex 结合现有代码认为短期用枚举字段更容易落地，也应确保后续可迁移到标签组模型，不要把业务逻辑散落在前端。

#### 4.3 后端必须校验系统预定义标签组互斥规则

前端可以限制用户只能点一个，但不能只依赖前端。

后端必须保证：

```text
同一知识点在 mastery 标签组中最多只能绑定一个标签
同一知识点在 importance 标签组中最多只能绑定一个标签
同一知识点在 purpose 标签组中可以绑定多个标签
普通标签组可以绑定多个标签
```

#### 4.4 原文片段必须保存快照

即使正文后续被修改，知识点仍然应该知道当初选中的原文是什么。

因此，KnowledgePointSource 至少应保存：

```text
sourceText
noteId
anchor 或 offset
contextBefore / contextAfter
```

这样后续才能支持：

1. 原文定位。
2. 原文失效提示。
3. 重新定位。
4. AI 出题时引用原文证据。
5. GraphRAG 构建时保留证据链。

#### 4.5 查询能力要面向未来 AI 出题设计

后续 AI 出题大概率需要这样的查询能力：

```text
按知识点 ID 出题
按普通标签出题
按掌握程度出题
按重要程度出题
按用途出题
按笔记 / 文件夹 / 知识空间出题
排除已掌握知识点
优先选择重点且未掌握知识点
```

所以当前后端查询不要只支持「根据 noteId 查列表」。应尽量设计成可组合查询。

#### 4.6 保留审计字段和软删除能力

建议所有核心实体都有：

```text
createdAt
updatedAt
deletedAt
createdBy / updatedBy（如项目已有）
```

知识点是长期学习资产，误删成本较高，建议使用软删除。

#### 4.7 保持用户与知识空间隔离

所有知识点、标签、原文片段、笔记关联都必须带有用户或知识空间维度。

后端查询必须保证：

```text
用户只能访问自己有权限的知识点
用户只能把原文片段绑定到自己可访问的笔记
标签筛选不能跨用户串数据
```

#### 4.8 为向量检索和 GraphRAG 预留空间

当前不必马上生成 embedding，但数据结构应便于后续扩展。

可预留方向：

```text
KnowledgePointEmbedding
SourceSnippetEmbedding
KnowledgePointRelation
KnowledgePointGraphEdge
```

未来可以将以下内容用于向量化：

1. 知识点标题。
2. 注释。
3. 原文片段。
4. 标签信息。
5. 所属笔记上下文。

### 5. 推荐后端实现顺序

#### 后端 V1：最小可用

```text

当前笔记知识点列表查询
普通标签绑定

```

#### 后端 V1.1：标签体系增强

```text

```

#### 后端 V1.2：面向复习和 AI 出题

```text

按掌握程度和重要程度统计

```


## 十四、最终交互闭环

理想使用流程：

```text
用户写 / 读笔记
↓
选中关键原文
↓
创建知识点
↓
补充注释
↓
添加普通标签
↓
设置掌握程度、重要程度、用途
↓
右侧卡片管理
↓
点击卡片可回到原文
↓
后续 AI 可以基于知识点出题、复习、建图谱和做 GraphRAG
```

一句话总结：

> **正文负责承载知识，知识点负责提炼知识，普通标签负责自由分类，系统预定义标签组负责学习状态管理，右侧面板负责当前笔记内的知识点操作。**

这个功能如果做好，会成为「学习加速器」中非常核心的知识抽取入口。它不是普通的右侧小功能，而是后续 AI 出题、错题分析、复习计划、知识图谱和 GraphRAG 的地基。
