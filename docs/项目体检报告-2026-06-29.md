## 1. 总体结论

1. **项目可正常运行**。`npm test` 全部通过(130 个 API 测试 + 109 个 Web 测试,共 239 个),`dev:api` 与 `dev:web` 均可启动,关键接口 `/api/health`、`/`、`/api/knowledge/notes`、`/api/knowledge/folders/tree` 均返回正确 JSON。
2. **无 P0 阻塞问题**。无启动失败、关键运行时错误、API 协议崩溃、严重安全漏洞。静态资源白名单 + 路径穿越检查、附件文件名 sanitize、JSON body 大小未做硬限但前端请求可控。

5. **建议进入下一阶段开发**。架构层面已经收口,新功能可按现有边界继续推进;但建议在进入新功能前先消化掉 5 个 P1/P2 项中的低风险局部修复。

## 4. 目录结构评价

### 4.1 归类清晰的目录

- `apps/api/src/modules/knowledge/{domain,application,infrastructure,http}` — DDD 四层结构清晰,domain 不依赖任何外部能力,application 只编排,infrastructure 只读写,http 只做路由调度。
- `apps/api/src/http/` — 纯通用 HTTP 边界,无业务知识库依赖(`storage-routes.js` 也仅是 storage 通用路由)。
- `apps/web/src/{app,controllers,server,services}` — 与 docs 完全一致,各目录职责分工明确。
- `apps/web/src/controllers/{navigation,editor,knowledge-point}/` — 业务 controller 进一步按子领域分目录,聚合入口文件保持 ≤ 250 行。
- `apps/web/lib/{navigation,editor,knowledge-points,search,sidebar,status,shell,notes,folders,tags}` — 按业务模块拆分子目录,渲染器和纯规则分离。
- `apps/web/lib/events/{aside-events/,...}` — 11 个事件 binder + 1 个 aside 内部子目录,各文件 ≤ 220 行,基本按 DOM 区域归类。
- `apps/web/lib/editor/milkdown/{commands,host,plugins,schema,table,utils}/` — Milkdown 集成子模块按职责切分,合理。

### 4.2 边界模糊/可优化点

- `apps/web/src/services/knowledge-api.js` 250 行,既包含 note / folder / tag / knowledge-point / space 5 类 endpoint 的封装,密度过高。
- `apps/web/lib/editor/milkdown/host/image-upload.js` 越过"lib 不做网络"边界。
- `apps/web/lib/editor/milkdown/` 下的 `table/` 已经有 8 个子文件,但 `table-handle-controller.js` 仍承担部分"事件接线 + 状态同步"职责,可视情况继续拆。
- `apps/web/styles/` 8 套主题 demo 占用了较多磁盘与文件树,这些 demo 与主应用完全隔离(从入口 `styles.css` 验证过),功能上无影响,但需要在 docs/项目结构导航.md 与 styles/README.md 之间明确"未来如何归档"。

### 4.3 拆分是否过度

无。整体看拆分遵循 docs/项目结构导航.md 的"按职责切分,不为了凑行数"原则。当前 20 个组件 CSS 文件、50+ 个 controller / 业务子模块,每个都有清晰职责命名。

### 4.4 建议调整

1. 把 `apps/web/src/services/knowledge-api.js` 按"notes / folders / tags / knowledge-points / spaces"拆 5 个子 service,在 `knowledge-api.js` 做再导出。
2. 把 `apps/web/lib/editor/milkdown/host/image-upload.js` 的 `fetch` 调用迁到 `src/services/knowledge-api.js`。
3. 把 `styles.css` 里残留的任务列表复选框样式迁入 `components/milkdown-content.css`。

## 5. API 评价

### 5.1 路径与 method

- 路径按资源命名,统一 `/api/knowledge/{notes,folders,tags,knowledge-points,knowledge-point-sources,knowledge-point-tag-groups,spaces,search}/...` 风格,清晰。
- method 使用合理:GET 查询、POST 创建/导入/恢复/收藏、PUT 替换 tag 集合、PATCH 部分更新、DELETE 删除/移出。例外:`POST /api/knowledge/notes/{id}/restore` 与 `POST /api/knowledge/notes/{id}/favorite` 是动作型而非创建,但当前选择可接受。
- 一个小的路径细节:删除笔记 tag 用 `DELETE /api/knowledge/notes/{id}/tags/{tagId}`,但新增单个 tag 用 `POST /api/knowledge/notes/{id}/tags` + body `tagId`。`{tagId}` 在 body 中是符合 REST 子资源约定的可接受变体,但前端 `knowledge-api.js` 中两个方法签名一致性可再确认。

### 5.2 参数与响应统一性

- 响应统一为 `{ data, error }`,`response.js` 中 `sendJson` 一处控制。
- 错误响应无 `code` 字段(后端规范中推荐 `error: {code, message}`),目前只用 message 字符串。
- 跨域:无 CORS 头配置。当前 `dev:web` 在 :3000、`dev:api` 在 :3001,前端通过 `apps/web/src/server/api-proxy.js` 走同源代理,生产部署需要重新评估 CORS。

### 5.3 错误处理

- `server.js` 用 `try/catch` 统一兜底返回 `400 { error: message }`,但其他状态码(401/403/409/500)没有显式区分,所有内部错误一律 400。
- `note-service.js`、`folder-service.js` 等 service 用 `throw new Error('message')` 抛出业务错误,message 直接面向前端。问题在于:错误信息可能泄漏内部结构(如"Note not found"是 ok 的,但若未来包含 `noteId` 就泄漏内部 ID 格式);同时没有业务错误码(只有字符串 message)。

### 5.4 安全性

- ✅ 静态资源白名单 + 路径穿越检查(`static-assets.js:27` `filePath.startsWith(rootDir)`)。
- ✅ 附件文件名 sanitize(`local-attachment-store.js:8-13`)。
- ⚠️ `parseBody` 没限制 `data` 长度(注释行可见 `data += chunk` 无限累加)。当前 dev 模式问题不大,生产部署需注意大 body 攻击。
- ✅ 服务端无 SQL 注入面(无 SQL);前端 API 调用全部 `encodeURIComponent`。
- ⚠️ 服务端 `appContext.modules.knowledge` 这条直接调用路径是 P1,见 §3.1。

### 5.5 前后端 service 一致性

- 前端 `apps/web/src/services/knowledge-api.js` 共 26 个方法,与后端 `knowledge-handlers.js` 中 33 个方法不完全 1:1(部分后端能力未在前端暴露,如 `assignTagToNotes` 批量、`createKnowledgePoint` 之外的部分)。无"旧接口残留":前端用的接口名与后端一致。
- ⚠️ 响应归一化层兼容性处理是技术债(见 P1 第 3 条)。

## 6. 前端规范评价

### 6.1 Token 与硬编码

- `apps/web/styles/tokens.css` 提供 35 个 design token(色阶、字阶、间距、圆角、阴影、面板尺寸),`tokens-css-shape.test.js` 静态校验过 baseline。
- 组件样式基本都通过 `var(--color-*)` / `var(--space-*)` 引用,未发现明显硬编码。
- `styles/components/milkdown-image.css` / `milkdown-table.css` 中存在少量 `rgba(60, 104, 255, 0.18)` 这类内联 alpha 颜色,这些是组件内"半透明叠加"惯例,严格说也是硬编码。可视情况下次清理时改为 `var(--color-accent-soft)` 系列 token。

### 6.2 原生弹窗

- `grep -rE "alert\s*\(|confirm\s*\(|prompt\s*\(" apps/web/src apps/web/lib` 未命中。前端无原生弹窗,符合规范。

### 6.3 独立滚动

- `shell-html.js` 与 CSS 中存在 `workspace-shell`、`icon-rail`、`aside` 等独立滚动容器,API 入口未发现 `window.scrollTo` 类的全局滚动。
- 顶部搜索/左侧导航/右侧栏/编辑器 4 个区域各自有 scroll 容器,样式在 `workspace-layout.css` 与 `responsive.css` 中。

### 6.4 事件/controller/lib/service 边界

- 事件层(`lib/events/*`)只接收 `deps` 并调用 `controller.xxx` 或纯函数,不再读 `state` 直接修改。
- controller(`src/controllers/*`)接收 `state` 快照 + `helpers`/`deps` 函数引用,只做编排,无 HTTP/IO。
- lib(`lib/*` 总体)主要是纯函数 + 渲染字符串 + 业务状态规则;有一处例外(`image-upload.js` 直接 fetch,见 P1 第 2 条)。
- service(`src/services/*`)做 API 请求封装 + 响应归一化,无业务判断。
- 主入口 `client.js` 151 行,只做对象创建 + 装配 + 事件绑定 + 运行时错误接入。✅

### 6.5 XSS 与 DOM 渲染

- `lib/navigation/context-menu-renderers.js` 与 `lib/editor/context-menu-renderers.js` 等渲染器使用 `escapeHtml` / `escapeAttribute` 工具。
- 控制器 `shell-controller.js`、`search-controller.js` 等使用 `innerHTML` 注入渲染字符串,但内容来自 lib 的纯渲染器,无外部直接拼字符串。
- ⚠️ `apps/web/src/styles.css` 中任务列表复选框使用 `border-left: 2px solid #fff` 这种硬编码 #fff,但 #fff 在 `tokens.css` 中没有等价 token,这里可视情况忽略(是任务列表勾选态的内层白线,非常小且为内容型)。
- ✅ 没有发现 `eval` / `Function(...)` / 直接 `document.write` 等高风险模式。

### 6.6 旧 React/Next 残留

- `apps/web/app/` 与 `apps/web/components/` 仍保留 React/Next 风格结构。docs 已明确标记为"非主运行路径";`shell-html.js` 不引用它们,运行时无影响。
- 建议:在 docs/项目结构导航.md 顶部加一句"当前主运行路径 = `src/client.js + src/main.js + lib/*`,`app/` 与 `components/` 为历史保留,改动前需确认入口",目前描述已存在于 §"### `apps/web/app` 与 `apps/web/components`"中,已满足。
