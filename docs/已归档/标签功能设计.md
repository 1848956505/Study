# 标签功能设计

> **顶部搜索栏负责“全局标签筛选”，右侧信息区负责“当前笔记标签管理”，左侧目录不再常驻展示标签。**

***

# 一、整体功能定位

标签功能分成两个入口：

| 区域       | 功能定位      | 主要操作             |
| :------- | :-------- | :--------------- |
| 顶部搜索栏    | 全局查找与标签筛选 | 选择多个标签、搜索笔记、清除筛选 |
| 右侧信息 Tab | 当前笔记标签管理  | 查看、添加、移除当前笔记标签   |
| <br />   | <br />    | <br />           |

这样职责非常清楚：

```
顶部：找东西
左侧：看目录
中间：写笔记
右侧：管理当前笔记属性

```

这套结构比把标签塞在左侧更干净，不会让左侧变成“功能大杂烩”。左侧终于可以松口气了，小目录树表示感谢。

***

# 二、顶部搜索栏：标签筛选的主入口

## 1. 搜索栏默认状态

默认状态保持你现在的风格：

```
⌕  搜索笔记、标签、附件、AI 结果

```

此时没有选中标签，搜索栏只承担普通搜索入口。

***

## 2. 搜索栏选中标签后的状态

用户在下拉面板中选择标签后，标签 Chip 直接嵌入搜索框：

```
⌕  [多模态 ×] [论文 ×]  搜索笔记、附件、AI 结果

```

这个设计非常关键。

因为标签筛选面板是隐藏的，如果不把已选标签显示在搜索框里，用户会忘记当前处于筛选状态，然后疑惑：“我的笔记怎么没了？”——不是笔记没了，是标签偷偷工作了。

***

## 3. 标签 Chip 样式

Chip 建议使用轻量胶囊样式：

```
[ 多模态 × ]
[ 论文 × ]
[ 待复习 × ]

```

视觉建议：

| 状态     | 样式                  |
| :----- | :------------------ |
| 普通已选标签 | 浅蓝背景、蓝色文字、圆角胶囊      |
| Hover  | 背景略深，显示可点击感         |
| 点击 ×   | 移除该标签筛选             |
| 标签过多   | 显示前 2~3 个，后面显示 `+3` |

例如：

```
⌕  [多模态 ×] [论文 ×] [+3]  搜索笔记...

```

避免搜索框被标签撑爆。

***

# 三、顶部搜索下拉面板设计

点击搜索栏后，向下弹出一个全局搜索面板。

建议布局如下：

```
┌──────────────────────────────────────────────┐
│ 搜索输入区                                    │
│ ⌕  [论文 ×] [多模态 ×]  输入关键词...          │
│                                              │
│ 已选标签                                      │
│ [论文 ×] [多模态 ×]                清除全部    │
│                                              │
│ 所有标签                                      │
│ [论文 5] [多模态 3] [重点 4] [待复习 2]        │
│ [AI 6] [计网 7] [英文阅读 2] [408 9]          │
│                                              │
│ 搜索结果                                      │
│ Vision Transformer 精读                       │
│ CLIP 论文笔记                                 │
└──────────────────────────────────────────────┘

```

这里“所有标签”区域：我希望设计成一种“标签云”的感觉

***

## 1. 面板顶部：搜索输入区

下拉面板内部可以继续显示一个搜索输入框，用来搜索笔记或筛选标签：

```
搜索笔记、标签、附件、AI 结果

```

当输入关键词时，可以同时过滤：

```
笔记标题
笔记内容
标签名称
附件名称
AI 结果

```

第一版可以先只做：

```
笔记标题 + 标签名称

```

不用一开始就把全文搜索、附件、AI 结果都做完。先做能跑的，再做能飞的。

***

## 2. 已选标签区

当用户选中标签后，面板内显示：

```
已选标签
[论文 ×] [多模态 ×]      清除全部

```

这里的作用是让用户可以集中管理筛选条件。

操作规则：

| 操作       | 效果          |
| :------- | :---------- |
| 点击标签上的 × | 移除单个筛选标签    |
| 点击清除全部   | 清空所有标签筛选    |
| 点击已选标签本体 | 可保持无操作，避免误触 |

***

# 四、标签筛选逻辑

第一版建议使用：

```
AND 逻辑：必须同时包含所有已选标签

```

例如：

```
选择：论文 + 多模态
结果：同时拥有「论文」和「多模态」标签的笔记

```

这个最适合学习资料库，因为用户筛标签通常是想越来越精确地缩小范围。

后续可以加一个切换：

```
匹配全部 / 匹配任一

```

对应：

```
tagMode: "all" | "any"

```

但第一版先固定为 `all` 就够了。

***

# 五、顶部搜索栏的前端状态设计

建议前端统一维护一个搜索筛选状态：

```
type SearchFilterState = {
  keyword: string;
  selectedTagIds: string[];
  tagMode: "all" | "any";
  scope: "all" | "currentFolder" | "currentSpace";
};

```

第一版可以简化为：

```
type SearchFilterState = {
  keyword: string;
  selectedTagIds: string[];
  tagMode: "all";
};

```

核心状态包括：

| 状态               | 说明               |
| :--------------- | :--------------- |
| `keyword`        | 搜索关键词            |
| `selectedTagIds` | 当前选中的标签          |
| `tagMode`        | 标签匹配模式，第一版固定 all |
| `isSearchOpen`   | 搜索下拉面板是否打开       |
| `tagOptions`     | 可选标签列表           |
| `searchResults`  | 搜索/筛选后的笔记结果      |

<br />

# 六、右侧标签区域：当前笔记标签管理

右侧标签区域继续保留在「信息」Tab 中。

## 1. 默认展示状态

```
标签                              3

[ 多模态 × ] [ ViT × ]
[ 论文精读 × ]

+ 添加标签

```

这里展示的是：

```
当前打开笔记已经绑定的标签

```

不是全局筛选。

***

## 2. 添加标签状态

点击 `+ 添加标签` 后展开：

```
标签                              3

[ 多模态 × ] [ ViT × ]
[ 论文精读 × ]

[ 输入标签名，搜索或新建 ]

可添加标签
[ 重点 ] [ 待复习 ] [ Transformer ]

```

交互规则：

| 操作        | 效果             |
| :-------- | :------------- |
| 点击已有标签    | 给当前笔记绑定该标签     |
| 输入新标签并回车  | 创建新标签，并绑定到当前笔记 |
| 点击当前标签 ×  | 仅从当前笔记移除标签     |
| 当前笔记已有的标签 | 不再出现在“可添加标签”里  |

***

## 3. 右侧标签删除规则

这里必须强调：

```
点击 × 只是从当前笔记移除标签，不删除全局标签。
当一个标签没有和任何一个笔记绑定时，这个标签需要自动销毁
```

例如：

```
当前笔记：[多模态 ×]

```

点击 `×` 后：

```
当前笔记不再绑定「多模态」
但「多模态」这个标签仍然存在于系统中

```

**真正标签销毁的逻辑：**

当该标签没有和任何一个笔记绑定时，自动销毁

***

# 八、右侧标签组件拆分

建议拆成：

```
NoteInfoPanel
└── NoteTagCard
    ├── CurrentNoteTagList
    ├── AddTagButton
    ├── TagPickerPopover
    └── TagSuggestionList

```

核心状态：

```
type NoteTagState = {
  noteId: string;
  currentTags: Tag[];
  availableTags: Tag[];
  isAdding: boolean;
  inputValue: string;
};

```

标签类型：

```
type Tag = {
  id: string;
  name: string;
  noteCount?: number;
  color?: string;
};

```

第一版 `color` 可以不做，后续再加标签颜色。

***

# 九、前端接口调用建议

虽然你说后端已有标签功能，但前端最好围绕这些接口组织。

## 1. 获取全局标签

```
GET /api/knowledge/tags?spaceId=space-demo

```

返回：

```
type TagDTO = {
  id: string;
  name: string;
  noteCount: number;
};

```

用于顶部搜索下拉面板。

***

## 2. 根据标签筛选笔记

```
GET /api/knowledge/notes?tagIds=tag1,tag2&tagMode=all&keyword=xxx

```

返回筛选后的笔记列表。

***

## 3. 获取当前笔记标签

```
GET /api/knowledge/notes/:noteId/tags

```

用于右侧信息 Tab。

***

## 4. 给当前笔记添加标签

```
POST /api/knowledge/notes/:noteId/tags

```

请求体：

```
{
  "tagId": "tag1"
}

```

或者支持新建：

```
{
  "name": "多模态"
}

```

***

## 5. 从当前笔记移除标签

```
DELETE /api/knowledge/notes/:noteId/tags/:tagId

```

注意这是：

```
删除绑定关系

```

不是删除标签本体。

***

# 十、UI 状态细节

## 1. 空状态

没有标签时：

```
暂无标签
你可以在右侧信息区为当前笔记添加标签

```

当前笔记没有标签时：

```
当前笔记暂无标签
+ 添加标签

```

***

## 2. 加载状态

顶部搜索下拉加载标签时：

```
正在加载标签...

```

右侧加载当前笔记标签时：

```
正在读取当前笔记标签...

```

可以用 skeleton 占位，不要让面板突然跳动。

***

## 3. 错误状态

接口失败时：

```
标签加载失败，点击重试

```

添加失败时：

```
添加标签失败，请稍后重试

```

删除失败时：

```
移除标签失败，请稍后重试

```

***

## 4. 标签过多处理

搜索框内已选标签过多时：

```
[论文 ×] [多模态 ×] [+3]

```

下拉面板里完整展示：

```
已选标签
[论文 ×] [多模态 ×] [重点 ×] [AI ×] [待复习 ×]

```

# 十二、开发实现顺序

建议你按这个顺序做：

```
第一步：实现右侧当前笔记标签展示
第二步：实现右侧添加 / 移除标签
第三步：实现顶部搜索栏内嵌已选标签 Chip
第四步：实现搜索栏下拉标签面板
第五步：实现多标签筛选笔记
第六步：实现搜索关键词 + 标签组合筛选
第七步：优化标签过多、省略、清除、空状态

```

我建议先做右侧，因为右侧是“标签产生入口”；然后再做顶部搜索栏筛选。逻辑上就是：

```
先能给笔记打标签
再能按标签找笔记

```

这个开发闭环最舒服。

# 十三、最终定稿方案

最终标签功能可以这样定：

```
1. 左侧目录栏不再展示标签云，只保留资料树、收藏、最近、回收站。

2. 顶部搜索栏作为全局搜索与标签筛选入口。

3. 用户点击搜索栏后，弹出搜索下拉面板，面板内展示：
   - 已选标签
   - 所有标签
   - 搜索结果

4. 用户选择标签后，标签 Chip 直接嵌入搜索框中。

5. 多个标签默认采用 AND 筛选逻辑。

6. 右侧信息 Tab 中展示当前笔记标签。

7. 右侧支持给当前笔记添加标签、移除标签。

8. 右侧的 × 只是移除当前笔记绑定，不删除全局标签。

9. 全局标签重命名、删除、合并等高级操作，后续放到独立“标签管理”弹窗中。

```

这套方案的核心优点是：**顶部负责找，右侧负责管，左侧负责导航**。\
三个区域职责不打架，后续扩展搜索、AI、知识点、出题都很好接。现在这个标签功能已经不是“加个小标签”了，而是知识库的第一套横向检索系统，挺关键，值得认真做。
