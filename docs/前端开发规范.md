# UI 开发规范

本文档记录学习加速器前端 UI 的开发原则与约束。它不规定具体界面长什么样，而是规定在开发 UI 时必须遵守的工程规范、结构边界和交互纪律。

## 1. 总体原则

- 页面优先服务内容，不优先堆叠装饰
- 外层壳层保持稳定，内容区域按模块演化
- 默认状态尽量简洁，复杂能力按需展开
- 能复用的就复用，能共享的就共享，避免重复造轮子
- 组件、样式、状态、布局职责要分开

## 2. 布局原则

- 整个应用页面固定，不允许把所有内容交给浏览器窗口统一滚动
- 顶栏、左侧导航栏、底栏应作为稳定壳层常驻显示
- 功能页面内部的各个区域可以独立滚动
- 内容区滚动应该局部化，避免一个滚动条同时承载所有信息
- 知识库等内容中心页面应优先保证正文区域可用空间

## 3. 区域滚动原则

- 左侧资源区、主内容区、右侧辅助区应可独立滚动
- 各区域应尽量拥有明确的滚动边界
- 页面级容器不应再承担具体内容滚动职责
- 长列表、目录、详情、附件、预览等内容应优先在本区域内滚动

## 4. 文案原则

- 功能区域只保留功能名称和必要数据，不放解释性长文案
- 不在模块里插入类似说明文、口号式提示或设计意图说明
- 空状态、辅助说明、引导文案应保持克制，能不用就不用
- 界面中优先展示可操作对象和当前数据，而不是描述性句子

## 5. 组件复用原则

- 重复出现两次以上的 UI 结构应优先抽成组件
- 按钮、标签、状态提示、列表项、面板容器、工具条应统一管理
- 页面只组合组件，不直接复制一整段相同结构
- 通用交互应沉淀为可复用组件或通用方法

## 6. 样式管理原则

- 颜色、间距、字号、圆角、阴影等必须集中管理
- 禁止在页面里随意硬编码视觉值，除非是局部且有明确原因
- 样式优先使用设计令牌、CSS 变量或统一 class
- 页面级样式只允许少量例外，不允许把整套视觉规则散落在页面里

## 7. 状态与逻辑原则

- 视图状态应尽量和业务数据解耦
- 一个页面的交互状态不要散落在多个地方互相拼接
- 视图更新应尽量通过明确的数据变更驱动
- 本地预览态和最终提交态要区分清楚

## 8. 模块边界原则

- 全局壳层只负责导航、搜索、状态和常驻操作
- 模块页面只负责自身内容和局部交互
- 知识库模块偏内容中心，其他模块可按自身特性调整布局
- 不要让某个模块的交互规则污染整套应用壳层

## 9. Markdown 编辑原则

- 默认以渲染后的内容为主，不常驻源码编辑器
- 源码编辑器应按需展开，不应该长期占据主视觉
- 格式化按钮应服务于内容编辑，不应变成装饰性工具条
- 预览与源码并排模式是当前优先级最高的源码查看方式

## 10. 响应式原则

- 桌面端优先
- 小屏幕优先保正文内容
- 辅助区、源码区、次要工具应优先折叠
- 不能因为适配小屏而破坏桌面端的内容中心结构

## 11. 交互克制原则

- 常用操作应靠近内容，但不抢占内容
- 底部栏可以放少量高价值动作，但不能演变成第二工具栏
- 过多的弹窗、浮层、提示条会破坏内容阅读节奏，应谨慎使用
- 动效和强调效果应帮助理解，不应喧宾夺主

## 12. 可维护性原则

- UI 结构要便于拆分和替换
- 组件命名要能表达职责，不要只表达位置
- 新功能应优先接入已有视觉体系，而不是另起一套
- 如果某处样式或结构开始重复，优先抽象，再继续写新页面
- UI 尽量不要使用浏览器自带的组件（如 `prompt`、`alert`、`confirm`、原生菜单和原生文件对话框），应优先使用项目内自定义输入框、弹层和菜单

## 13. 开发检查清单

每次新增或修改 UI 时，至少检查以下问题：

1. 页面是否仍然保持壳层固定、区域独立滚动
2. 是否存在不必要的解释性文案
3. 是否有可复用组件可以抽出来
4. 是否出现了硬编码视觉值
5. 是否破坏了全局壳层和模块页面的边界
6. 是否让内容区被无关信息挤压

## 14. 适用范围

本规范适用于：

- Web 工作台壳层
- 知识库模块页面
- 后续试卷、AI、任务、复盘等模块
- 共享 UI 组件与样式层

本规范不限制每个模块具体长什么样，只限制它们应该怎样被开发、怎样保持一致、怎样避免失控。



# 后端开发规范

本文档记录学习加速器后端的开发原则与约束。它不规定具体功能如何实现，而是规定在开发后端时必须遵守的工程规范、模块边界、接口纪律和数据安全要求。

## 1. 总体原则

- 后端优先保证数据正确性、接口稳定性和业务边界清晰
- 功能按模块组织，模块内部高内聚，模块之间低耦合
- 接口、数据、权限、日志、错误处理应有统一规范
- 新功能应优先接入已有后端结构，不随意另起一套实现方式
- 业务逻辑不应散落在路由层、控制器层或数据库操作中
- 能复用的服务、校验、工具、错误处理应统一沉淀

## 2. 分层原则

后端代码应按照职责分层，避免一个文件承担过多职责。

推荐分层如下：

- 路由层：只负责接收请求、绑定路径、调用控制器
- 控制器层：负责解析请求、调用服务、返回响应
- 服务层：负责业务逻辑、业务规则、事务组织
- 数据访问层：负责数据库查询、数据持久化
- 模型层：负责数据结构、实体定义、字段约束
- 工具层：负责通用函数，不包含具体业务逻辑

禁止将复杂业务逻辑直接写在路由层或控制器层中。

## 3. 模块边界原则

- 每个功能模块应拥有相对独立的目录结构
- 模块内部可以包含 route、controller、service、repository、model、types 等内容
- 模块之间不应随意互相调用内部实现
- 跨模块调用应通过明确的 service 接口完成
- 不允许一个模块直接修改另一个模块的核心数据，除非有明确业务入口
- 公共能力应放入 shared、common 或 core 层，不应复制到多个模块中

例如：

```txt
backend/
  src/
    modules/
      knowledge/
        knowledge.routes.ts
        knowledge.controller.ts
        knowledge.service.ts
        knowledge.repository.ts
        knowledge.types.ts
      task/
      ai/
      review/
    shared/
      errors/
      response/
      validation/
      logger/
      database/
      auth/
```

## 4. API 接口原则

- API 路径应稳定、语义清晰、风格统一
- 接口应围绕资源设计，而不是围绕页面设计
- 请求参数、响应结构、错误格式应保持一致
- 不同模块不得各自定义一套响应格式
- 接口返回的数据应服务前端使用，但不应暴露无关内部字段
- 接口命名应避免临时性、测试性、含糊性名称

推荐路径风格：

```txt
GET    /api/knowledge/spaces
POST   /api/knowledge/spaces
GET    /api/knowledge/notes/:noteId
PUT    /api/knowledge/notes/:noteId
DELETE /api/knowledge/notes/:noteId
```

不推荐：

```txt
/api/getAllData
/api/saveSomething
/api/test
/api/doNoteAction
```

## 5. 响应结构原则

所有接口应使用统一响应结构。

成功响应推荐：

```json
{
  "success": true,
  "data": {},
  "message": ""
}
```

列表响应推荐：

```json
{
  "success": true,
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "pageSize": 20
  },
  "message": ""
}
```

失败响应推荐：

```json
{
  "success": false,
  "error": {
    "code": "NOTE_NOT_FOUND",
    "message": "笔记不存在"
  }
}
```

禁止不同接口返回完全不同的格式。

## 6. 错误处理原则

- 后端应统一处理错误，不应在各处随意返回错误对象
- 业务错误、参数错误、权限错误、系统错误应区分清楚
- 错误信息应对前端友好，但不暴露数据库、服务器路径、堆栈等内部信息
- 不允许直接把原始异常完整返回给前端
- 可预期错误使用业务错误码，不可预期错误记录日志后返回通用错误

常见错误类型：

```txt
VALIDATION_ERROR      参数错误
UNAUTHORIZED          未登录
FORBIDDEN             无权限
RESOURCE_NOT_FOUND    资源不存在
CONFLICT              数据冲突
INTERNAL_ERROR        服务器内部错误
```

## 7. 参数校验原则

- 所有外部输入必须经过校验
- 路径参数、查询参数、请求体都应校验
- 不允许直接信任前端传来的数据
- 创建和更新接口应区分校验规则
- 校验逻辑应集中管理，不应散落在业务代码中

需要校验的内容包括：

- 字段是否存在
- 字段类型是否正确
- 字符串长度
- 数值范围
- 枚举值是否合法
- ID 是否符合格式
- 业务上是否允许该操作

## 8. 数据库设计原则

- 数据库表结构应围绕业务实体设计，不围绕前端页面设计
- 表名、字段名应语义明确、风格统一
- 重要表应包含创建时间、更新时间
- 需要软删除的业务数据应使用 deletedAt 或状态字段
- 不应随意物理删除重要数据
- 外键关系、唯一约束、索引应根据业务需要明确设计
- 数据库字段不应保存可以轻易计算出来的冗余数据，除非有性能原因

推荐基础字段：

```txt
id
created_at
updated_at
deleted_at
created_by
updated_by
```

## 9. 数据访问原则

- 数据库访问应集中在 repository 或 data access 层
- 服务层不应直接拼接复杂 SQL
- 不允许在控制器层直接访问数据库
- 查询条件应明确，不应写无边界查询
- 列表接口必须考虑分页、排序和筛选
- 删除、更新操作必须有明确条件，避免误操作全表数据

禁止危险操作：

```sql
DELETE FROM notes;
UPDATE notes SET status = 'deleted';
```

除非有明确迁移脚本或管理操作，并经过确认。

## 10. 事务原则

- 涉及多个数据库写操作的业务必须考虑事务
- 同一个业务动作中的数据变更应保持原子性
- 事务边界应放在服务层，而不是控制器层
- 事务内不应执行耗时的外部调用
- 事务失败后应保证数据不处于半完成状态

例如：

- 创建笔记并写入初始版本，需要事务
- 删除文件夹并移动或清理子笔记，需要事务
- 导入知识库数据，需要事务或分批回滚策略

## 11. 权限与认证原则

- 用户身份认证应集中处理，不应每个接口单独实现
- 权限判断应与业务操作绑定
- 不允许只依赖前端隐藏按钮来控制权限
- 敏感操作必须在后端再次校验权限
- 普通用户、管理员、系统任务应区分操作边界
- 后续多用户场景中，所有资源都应考虑归属关系

典型权限问题：

- 用户是否可以查看该空间
- 用户是否可以编辑该笔记
- 用户是否可以删除该文件夹
- 用户是否可以导入或导出数据
- 用户是否可以调用 AI 能力

## 12. 配置管理原则

- 端口、数据库连接、密钥、模型地址等配置不得硬编码
- 环境变量应集中读取和校验
- 开发环境、测试环境、生产环境配置应区分
- 敏感信息不得提交到代码仓库
- 默认配置应保证本地开发可以快速启动

配置示例：

```txt
APP_PORT
DATABASE_URL
JWT_SECRET
UPLOAD_DIR
AI_API_BASE_URL
LOG_LEVEL
```

## 13. 日志原则

- 后端应记录关键操作日志和错误日志
- 日志应帮助定位问题，而不是制造噪音
- 不应在日志中输出密码、token、密钥等敏感信息
- 业务关键操作应记录操作者、操作对象、操作时间和结果
- 系统错误应记录足够的上下文，方便排查

应记录的内容：

- 服务启动与关闭
- 数据库连接异常
- 登录与权限失败
- 创建、删除、导入、导出等重要操作
- 第三方服务调用失败
- 未捕获异常

## 14. 文件与资源管理原则

- 上传文件、导入文件、导出文件应有统一管理策略
- 文件元数据应入库，文件本体应存储在明确目录或对象存储中
- 文件名、扩展名、大小、类型必须校验
- 不允许使用用户上传的原始文件名直接作为服务器文件名
- 删除业务数据时应考虑关联文件的清理策略
- 导入导出功能应考虑版本兼容和错误恢复

## 15. AI 调用原则

- AI 能力应作为独立服务封装，不应散落在业务代码中
- AI 输入和输出应有明确结构
- AI 生成结果不能直接写入正式数据，除非用户确认
- 本地预览态、草稿态、正式提交态必须区分
- AI 调用失败不应影响核心数据安全
- 应记录必要的调用状态，但不记录敏感内容

例如：

```txt
用户笔记内容 -> AI 服务 -> 生成摘要预览 -> 用户确认 -> 写入笔记扩展字段
```

不推荐：

```txt
用户点击按钮 -> AI 返回结果 -> 直接覆盖原笔记
```

## 16. 本地数据与导入导出原则

- 本项目优先支持本地数据安全保存
- 导出数据应包含必要的版本信息
- 导入数据前应校验格式、版本和完整性
- 导入过程不应破坏现有数据
- 大规模导入应支持错误报告
- 一键打包和一键导入应有清晰的数据边界

导出包建议包含：

```txt
metadata.json
database_dump 或 structured_data.json
attachments/
version.txt
```

## 17. 测试原则

- 核心业务逻辑应优先编写单元测试
- 重要接口应编写接口测试
- 数据库相关逻辑应测试正常情况和异常情况
- 删除、导入、导出、权限判断等高风险功能必须测试
- 测试数据应独立，不应依赖真实用户数据
- 修复 bug 后应补充对应测试，避免再次出现

## 18. 性能原则

- 列表接口必须分页
- 搜索、目录树、标签聚合等接口应考虑索引
- 不应一次性返回大量正文、附件或历史版本
- 大文件处理应避免阻塞主请求
- 高频接口应避免重复查询
- 必要时可以引入缓存，但缓存不能破坏数据一致性

## 19. 安全原则

- 所有用户输入都必须校验和转义
- 禁止拼接未处理的 SQL
- 禁止把 token、密码、密钥写入日志
- 文件上传必须限制类型和大小
- 下载文件必须校验权限
- 管理类接口必须有额外权限控制
- 后端不能信任前端传来的用户身份、角色和权限字段

## 20. 命名原则

- 文件名、函数名、变量名应表达业务职责
- 不使用含糊名称，如 data、info、handle、doSomething
- service 方法应表达业务动作
- repository 方法应表达数据访问意图
- 错误码应统一大写，并能表达具体错误原因

推荐命名：

```txt
createNote
updateNoteContent
deleteFolderAndMoveNotes
findNotesBySpaceId
validateNotePayload
```

不推荐命名：

```txt
handleData
doUpdate
process
save
testApi
```

## 21. 可维护性原则

- 新功能应优先复用已有模块、服务和工具
- 重复逻辑出现两次以上应考虑抽象
- 不应为了快速实现把临时逻辑写进核心流程
- 删除旧代码时应同步清理路由、类型、测试和文档
- 模块变复杂时应主动拆分，而不是继续堆文件
- 重要业务规则应有注释说明原因，而不是只解释代码表面行为

## 22. 开发检查清单

每次新增或修改后端功能时，至少检查以下问题：

1. 是否保持了 route、controller、service、repository 的职责边界
2. 是否存在业务逻辑写在路由层或控制器层的问题
3. 是否使用了统一响应结构
4. 是否使用了统一错误处理
5. 是否对所有外部输入做了参数校验
6. 是否考虑了权限校验
7. 是否存在硬编码配置
8. 是否存在无分页的大列表接口
9. 是否存在危险的更新或删除操作
10. 是否需要事务
11. 是否记录了必要日志
12. 是否泄露了敏感信息
13. 是否破坏了模块边界
14. 是否有重复逻辑可以抽象
15. 是否需要补充测试

## 23. 适用范围

本规范适用于：

- 后端 API 服务
- 数据库访问层
- 知识库模块
- 任务模块
- AI 模块
- 复盘模块
- 导入导出模块
- 用户与权限模块
- 共享服务、工具函数与基础设施代码

本规范不限制具体技术实现，只限制后端应该如何组织、如何协作、如何保证数据正确性和系统可维护性。
